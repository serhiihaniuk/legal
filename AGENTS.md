# AGENTS.md — Legalizacja: mapa prawa

## Project context

This repository contains a multi-page educational React application for a person working with the legalization of foreigners in Poland.

The product teaches the user how to:

- read and understand `Kodeks postępowania administracyjnego` (KPA);
- navigate `ustawa o cudzoziemcach` and rules concerning the employment of foreigners;
- translate Polish legal terminology into practical reasoning;
- move from facts to a legal concept, provision, evidence, deadline, consequence, and remedy;
- determine the actual purpose of stay (`cel pobytu`) instead of relying only on the name of an application;
- read `wezwania`, `decyzje`, `postanowienia`, and appeal instructions;
- see the whole system as a mind map and then deepen individual branches with real examples.

The primary user communicates in Ukrainian and works with Polish legal documents. Polish names of acts, institutions, procedures, and legal terms must therefore remain in Polish, while explanations should be written in clear Ukrainian.

## Product shape

The current product uses Vite, React, TypeScript, React Router, Tailwind CSS v4, and the official shadcn `base-vega` starter:

- `app/routes.ts` — application routes;
- `app/routes/` — route-level pages;
- `app/components/` — product components;
- `app/components/ui/` — shadcn primitives generated by the official CLI;
- `app/data/legal-data.ts` — core educational content and official source links;
- `app/data/kpa-guide-data.ts` — the deep KPA curriculum;
- `app/data/kpa-article-index.ts` — the 306-entry article index;
- `legal-corpus/documents/` — versioned ELI source configurations;
- `scripts/legal-corpus/build-document.mjs` — Node-only ELI/PDF ingestion pipeline;
- `scripts/legal-corpus/workflow.mjs` — deterministic prepare/diff/validate/promote workflow;
- `app/data/legal-corpus/` and `public/legal-sources/` — generated provision data and local PDFs;
- `app/data/legal-library/` — typed document/provision contracts and the canonical query/resolution module;
- `app/routes/law*.tsx` — multi-document catalog, provision reader, and KPA compatibility routes;
- `app/components/kpa-articles-content.tsx` — legacy-compatible KPA selector and reader;
- `app/data/legal-types.ts` — the content schema;
- `app/data/legal-index.ts` — recursive indexing;
- `app/app.css` — Tailwind v4 theme tokens and global accessibility rules;
- `DESIGN.md` — design and UX source of truth.

Use the existing stack. Do not reintroduce global browser data, inline data scripts, or parallel legacy implementations.

Build the KPA corpus with `npm run corpus:kpa` or any schema-v2 edition with `npm run corpus:build -- <config>`. Regenerate typed registries with `npm run corpus:generate`; use `corpus:prepare`, `corpus:validate`, and explicit `corpus:promote` for edition changes. The importer must remain Node-only so the project does not require a second runtime. Generated provision text is a locator and learning aid; the source reader must retain the official ELI link for legal verification. New editions never become current merely because they were fetched or built.

## Core learning model

Every topic should reinforce this sequence:

```text
Fakt
→ Pojęcie prawne
→ Przepis i jego warunki
→ Dowód dla każdego warunku
→ Czynność i termin
→ Skutek
→ Środek zaskarżenia
```

Do not teach provisions as isolated article numbers. Explain what a provision changes in a case and how it connects to the surrounding procedure.

## Content scope

The main map should cover, at an orientation level:

1. legal reasoning and reading a legal norm;
2. KPA as the procedural framework;
3. entry and current basis of stay;
4. purposes of temporary residence;
5. permanent residence and long-term EU resident status;
6. access to the Polish labour market;
7. documents and evidence;
8. `wezwania`, decisions, appeals, inactivity, and judicial review;
9. practical anonymized case studies.

Detailed international protection, citizenship, criminal law, and tax law are outside the first-version scope unless the user asks to add them.

The project is an educational tool, not an automated decision system or a substitute for individual legal advice.

## Legal-source policy

Legal accuracy is a high-risk and time-sensitive part of this project.

- The visible legal-state date is currently `14.07.2026`.
- Before changing legal content, verify the current law using official sources. Do not rely on memory alone.
- Prefer primary sources in this order:
  1. `eli.gov.pl`, Sejm ELI, and `Dziennik Ustaw`;
  2. official `gov.pl` pages of UdSC, MRPiPS, MSWiA, or another competent authority;
  3. official court databases and published judgments;
  4. secondary commentary only as supporting context, never as the sole basis for a legal claim.
- Check the status of the act, consolidated text, amending acts, entry-into-force dates, and transitional provisions.
- Match the applicable rule to the relevant date of filing or factual event.
- Distinguish clearly between:
  - what the statute expressly says;
  - official administrative guidance;
  - case-law interpretation;
  - a practical inference or checklist.
- Do not describe a future amendment as current law.
- Use exact article references and avoid invented article ranges.
- Keep official links in each relevant `legal-data.ts` node.
- The visible legal-state date is derived from `legalData.updatedAt`; update that single field after verifying the new baseline.

Key official acts currently represented in the map include:

- KPA — `Dz.U. 2025 poz. 1691`;
- `ustawa o cudzoziemcach` — `Dz.U. 2025 poz. 1079`, with later amendments;
- `ustawa z 20 marca 2025 r. o warunkach dopuszczalności powierzania pracy cudzoziemcom` — `Dz.U. 2025 poz. 621`;
- `Prawo o postępowaniu przed sądami administracyjnymi` — `Dz.U. 2026 poz. 143`.

These citations describe the current project baseline, not a guarantee that they remain current in a future editing session.

## Rules for explanations

Use a calm, concrete, educational voice.

- Preserve the Polish legal term: `wezwanie`, `brak formalny`, `odwołanie`, `pracodawca użytkownik`.
- Explain it immediately in simple Ukrainian.
- Prefer short sentences and practical questions.
- Avoid literal translation when it hides the procedural meaning.
- Explain the difference between similar concepts, especially:
  - legal stay vs right to work;
  - `brak formalny` vs failure to prove a substantive condition;
  - `decyzja` vs `postanowienie`;
  - `odwołanie` vs `zażalenie`;
  - `wznowienie` vs `stwierdzenie nieważności`;
  - `outsourcing procesowy` vs provision of workers / temporary work.
- State uncertainty where the result depends on the full facts, actual organization of work, transitional rules, or local-office practice.
- Never imply that one contract clause, document, or template automatically guarantees a legal outcome.

## Mind-map data contract

All educational nodes live in `src/data/legal-data.ts` under `legalData.groups` and must satisfy `LegalAtlasData`.

Each substantive node should use as many of these fields as are relevant:

```js
{
  id: "stable-unique-id",
  title: "Ukrainian learner-facing title",
  polish: "Polish legal term or article range",
  summary: "Plain-language explanation",
  why: "What this changes in a real case",
  checkpoints: ["Questions the learner should ask"],
  steps: ["Ordered practical analysis steps"],
  documents: ["Relevant documents or evidence"],
  sources: [{ label, url, note }],
  related: ["existing-node-id"],
  children: []
}
```

Requirements:

- every `id` must be unique and stable;
- every `related` value must point to an existing node;
- every source URL must use HTTPS and lead to an authoritative page;
- a child belongs under the concept that explains its role;
- do not create duplicate nodes just to mention the same article in several places; link with `related` where possible;
- broad nodes provide orientation, while child nodes add detail.

## Design and interaction rules

Follow `DESIGN.md` for product and visual decisions.

- Keep the legal-atlas character: warm paper, dark navy, restrained amber and sage accents.
- Preserve the overview-first hierarchy.
- The map and detail panel must remain usable with a keyboard.
- Do not encode meaning with color alone.
- Maintain readable contrast and visible focus states.
- Desktop uses map and detail panel side by side; smaller screens stack them.
- Learning screens use a documentation layout: global routes in a compact top header, local sections on the left, one naturally scrolling document in the center, and a short on-page table of contents on the right.
- Do not add search controls. Navigation uses the global menu, local indexes, mobile selectors, and previous/next controls.
- Ordinary internal movement belongs to navigation controls, not ad hoc prose links. Explicit typed references to a legal provision or evidence document may appear inline. All `prose`, `reference-section`, and `provision-page` contexts use the same quiet inherited text color with a visible underline, hover/focus state, and preview; do not use navigation-blue merely because a legal reference is clickable. Keep the semantic context explicit where appropriate, never infer a link from a title or bare article number, and keep external official sources as explicit source links.
- Do not create simultaneously scrolling navigation and reading panels. Large selectors such as the 306-article catalog open in a shadcn `Sheet`, then return the learner to the central document.
- On mobile, global and dense local navigation move into shadcn `Sheet` components or compact native selectors with clear labels.
- Keep touch targets usable at 360 px viewport width.
- Respect `prefers-reduced-motion`.
- Avoid decorative additions that compete with reading.

## Case-study safety

Real `wezwania`, decisions, contracts, and client documents may contain sensitive personal or business data.

- Use only anonymized examples.
- Remove names, dates of birth, passport numbers, case numbers, addresses, contact data, signatures, bank data, and other identifiers.
- Do not commit original client documents to the repository.
- Convert a case into a generalized learning pattern: facts, legal question, applicable rules, evidence, risks, and possible procedural paths.
- Do not present a model response as guaranteed to be sufficient for every authority or factual situation.

## Change workflow

For content changes:

1. identify the exact learner question;
2. locate the node or branch where the concept belongs;
3. verify current official legal sources;
4. update the explanation and source link together;
5. add cross-links instead of duplicating content;
6. update the legal-state date when the verified baseline changes;
7. verify the site behavior.

For interface changes:

1. review `DESIGN.md`;
2. reuse existing shadcn components and semantic Tailwind tokens before creating new primitives;
3. keep route-level pages focused and product components reusable;
4. keep diffs small and accessible;
5. test navigation, deep links, search, filters, disclosure controls, detail selection, source links, and responsive behavior.

## Verification before completion

Run at least:

```text
npm run typecheck
npm run build
```

Also verify:

- the page renders meaningful content;
- there are no browser console errors;
- search returns the expected branch;
- selecting a node replaces the placeholder with its details;
- every `related` node exists;
- source links use official URLs;
- there is no horizontal overflow at desktop and mobile widths;
- direct navigation to each React Router URL renders the expected page;
- the production build completes without errors.

For legal-content work, report the official sources checked and the date of the legal state. If a source could not be verified, state that gap instead of presenting the content as certain.
